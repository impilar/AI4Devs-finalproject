> Como usuario final, quiero ver un mensaje claro cuando aún no tengo notas, para saber cómo empezar a capturar contenido.

## Criterios de aceptación (Gherkin)

```gherkin
Feature: Estado vacío del listado

  Scenario: Usuario sin notas ve mensaje orientativo
    Given que no tengo ninguna nota creada
    When abro la aplicación
    Then veo un mensaje que indica que aún no hay notas
    And veo un botón o enlace "Crear nota"

  Scenario: El estado vacío no muestra pantalla en blanco
    Given que el listado de notas está vacío
    When cargo la pantalla principal
    Then no veo una pantalla completamente en blanco sin contexto
```

## Detalle de implementación (historia)

### Resumen técnico

Cuando `GET /api/v1/notas` devuelve `data: []` sin filtros activos (`etiqueta`, búsqueda), `NoteList` muestra `EmptyState` con mensaje orientativo y CTA «Crear nota» hacia `/notas/nueva`. El backend ya soporta listado vacío en MVP; V1 formaliza el contrato, el copy alineado con Gherkin y la cobertura E2E. Los estados vacíos de **filtro por etiqueta** y **búsqueda sin resultados** son historias distintas (US-009 / US-014) y no deben mezclarse con el empty state de biblioteca vacía.

### API involucrada

| Método | Ruta | Uso en esta historia |
|--------|------|----------------------|
| GET | `/api/v1/notas` | Listado sin notas → `200` con `data: []` |
| GET | `/api/v1/health` | Smoke test en E2E (stack levantado) |

**Respuesta 200 (biblioteca vacía):**

```json
{
  "data": [],
  "meta": { "total": 0 }
}
```

**No aplicar en US-003:** query `?etiqueta=` ni flujo de búsqueda (`/buscar`).

### Contratos (DTOs / campos)

| Campo JSON | Tipo | En empty state | Notas |
|------------|------|----------------|-------|
| `data` | `NotaResumen[]` | `[]` | Sin error 404 ni 204 |
| `meta.total` | number | `0` | Coherente con longitud de `data` |

### TASK-009 — [BE] Respuesta consistente GET /notas con array vacío

| Campo | Valor |
|-------|-------|
| **Tipo** | Backend |
| **Release** | V1 |
| **Depende de** | US-001 / TASK-001 (endpoint existente) |

**Objetivo:** Garantizar que la API de listado devuelve envelope consistente cuando no hay filas en `notas`.

**Archivos a crear o modificar:**

- `src/backend/src/controllers/nota.controller.ts` — `listNotas` sin rama especial que devuelva 404
- `src/backend/src/services/nota.service.ts` — `list` retorna `{ data: [], meta: { total: 0 } }` si vacío
- `src/backend/src/repositories/nota.repository.ts` — `findAll` con `findMany` sin joins innecesarios en listado base

**Pasos de implementación:**

3. Añadir o reforzar test de integración: BD vacía → `GET /api/v1/notas` → `200`, `data: []`, `meta.total: 0`.
4. Documentar en comentario de test que este contrato alimenta US-003 empty state.

**Criterios de done (verificables):**

- [ ] `GET /api/v1/notas` con cero notas → HTTP 200 (no 204, no 404)
- [ ] Body `{ data: [], meta: { total: 0 } }` en camelCase
- [ ] Sin excepciones Prisma ni 500 por tabla vacía
- [ ] Test de integración dedicado en verde

**Tests:**

| Tipo | Ubicación | Qué valida |
|------|-----------|------------|
| Integración | `tests/integration/api/notas.list.test.ts` | Lista vacía, envelope y status 200 |
