## Detalle de implementación (historia)

### Resumen técnico

`GET /api/v1/etiquetas` deja de devolver solo nombres (`string[]`) y pasa a devolver objetos `{ id, name, count }` ordenados alfabéticamente por `name`, incluyendo etiquetas con `count: 0`. En `HomePage`, el sidebar sustituye o extiende `TagFilter` por `TagCatalog`, mostrando cada etiqueta como «nombre (N)». Al pulsar una fila, se activa el mismo flujo de filtro que US-009 (`activeTag` + `GET /notas?etiqueta=`). El conteo se calcula con agregación SQL/Prisma sobre `nota_etiqueta` sin vista materializada (volumen < 500 notas).

### API involucrada

| Método | Ruta | Uso en esta historia |
|--------|------|----------------------|
| GET | `/api/v1/etiquetas` | Catálogo con conteo por etiqueta |
| GET | `/api/v1/notas?etiqueta={nombre}` | Filtro al seleccionar en catálogo (US-009) |

**Respuesta 200 (catálogo):**

```json
{
  "data": [
    { "id": "uuid", "name": "archivo", "count": 0 },
    { "id": "uuid", "name": "ideas", "count": 5 }
  ]
}
```

**Breaking change controlado:** el endpoint ya existe en MVP devolviendo `string[]`; V2+ migra el contrato a objetos. Actualizar `listEtiquetas` en FE y tests de integración existentes.

### Contratos (DTOs / campos)

| Campo JSON | Tipo | Obligatorio | Notas |
|------------|------|-------------|-------|
| `data[].id` | UUID | sí | Para futuras operaciones; no requerido en UI de filtro (usa `name`) |
| `data[].name` | string | sí | Nombre único de etiqueta |
| `data[].count` | number | sí | Entero ≥ 0; notas distintas asociadas vía `nota_etiqueta` |
