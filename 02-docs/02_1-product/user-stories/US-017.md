## Detalle de implementación (historia)

### Resumen técnico

Se introduce la tabla `nota_backlink` (origen → destino) con FK a `notas`. En modo edición, el usuario selecciona una nota destino con `NoteLinkPicker`; al guardar se llama `POST /api/v1/notas/:origenId/backlinks`. En vista detalle, la nota origen muestra enlaces salientes clicables; la nota destino muestra la sección «Notas que enlazan aquí» con backlinks entrantes vía `GET /api/v1/notas/:id/backlinks/entrantes`. No incluye grafo visual, plugins ni eliminación de enlaces (fuera de Gherkin V2+).

### API involucrada

| Método | Ruta | Uso en esta historia |
|--------|------|----------------------|
| POST | `/api/v1/notas/:id/backlinks` | Crear enlace saliente desde nota origen |
| GET | `/api/v1/notas/:id/backlinks/salientes` | Listar notas enlazadas desde la actual (vista detalle origen) |
| GET | `/api/v1/notas/:id/backlinks/entrantes` | Listar notas que enlazan a la actual (panel entrantes) |
| GET | `/api/v1/notas` | Poblar selector en edición (excluir nota actual) |
| GET | `/api/v1/notas/:id` | Detalle tras guardar |

**Propuesta API** (gap LLD — alineada con HLD-v1 §12.2; confirmar en OpenSpec design):

**POST** `/api/v1/notas/:id/backlinks`

```json
{ "destinoId": "uuid-de-nota-destino" }
```

Respuesta `201`:

```json
{
  "data": {
    "origenId": "uuid",
    "destinoId": "uuid",
    "destino": { "id": "uuid", "title": "Investigación de mercado" }
  }
}
```

**GET** `/api/v1/notas/:id/backlinks/salientes` y **GET** `.../entrantes`

```json
{
  "data": [
    { "id": "uuid", "title": "Plan Q3" }
  ]
}
```

**Errores:**

| Código | Condición |
|--------|-----------|
| `400 VALIDATION_ERROR` | `destinoId` inválido, origen === destino, enlace duplicado |
| `404 NOT_FOUND` | Nota origen o destino no existe |

### Contratos (DTOs / campos)

| Campo JSON | Tipo | Notas |
|------------|------|-------|
| `destinoId` | UUID | Body POST; nota destino del enlace |
| `data[].id` | UUID | Nota relacionada en listados salientes/entrantes |
| `data[].title` | string | Título para enlace clicable en UI |

### TASK-067 — [DB] Tabla nota_backlink (origen_id, destino_id)

| Campo | Valor |
|-------|-------|
| **Tipo** | Database |
| **Release** | V2+ |
| **Depende de** | US-001 (tabla `notas`) |

**Objetivo:** Persistir relaciones dirigidas entre notas con integridad referencial.

**Archivos a crear o modificar:**

- `src/backend/prisma/schema.prisma` — model `NotaBacklink`
- `src/backend/prisma/migrations/YYYYMMDD_add_nota_backlink/migration.sql`
- `02-docs/02_2-architecture/data-model/logical-model-v1.md` — opcional: promover §11.2 a implementado

**Pasos de implementación:**

1. Añadir model Prisma:

```prisma
model NotaBacklink {
  origenId   String @map("origen_id") @db.Uuid
  destinoId  String @map("destino_id") @db.Uuid
  origen     Nota   @relation("BacklinksSalientes", fields: [origenId], references: [id], onDelete: Cascade)
  destino    Nota   @relation("BacklinksEntrantes", fields: [destinoId], references: [id], onDelete: Cascade)

  @@id([origenId, destinoId])
  @@index([destinoId], map: "idx_nota_backlink_destino")
  @@index([origenId], map: "idx_nota_backlink_origen")
  @@map("nota_backlink")
}
```

2. Añadir relaciones inversas en model `Nota`.
3. Generar y aplicar migración.
4. Regla de negocio en servicio (no CHECK en BD): `origenId !== destinoId`.
5. UNIQUE compuesta `(origen_id, destino_id)` vía `@@id`.

**Criterios de done (verificables):**

- [ ] Tabla `nota_backlink` existe en PostgreSQL
- [ ] FK `origen_id` y `destino_id` → `notas.id` ON DELETE CASCADE
- [ ] Índices `idx_nota_backlink_origen` y `idx_nota_backlink_destino` creados
- [ ] `npx prisma migrate deploy` exitoso en entorno limpio

**Tests:**

| Campo | Valor |

| **Tipo** | Backend |
| **Release** | V2+ |
| **Depende de** | TASK-067 |

**Objetivo:** Exponer endpoints REST para crear enlaces entre notas y consultar salientes y entrantes.

**Archivos a crear o modificar:**

- `src/backend/src/repositories/backlink.repository.ts` — `create`, `findSalientes`, `findEntrantes`
- `src/backend/src/services/backlink.service.ts` — reglas: no self-link, no duplicado, notas existen
- `src/backend/src/controllers/backlink.controller.ts` — handlers HTTP
- `src/backend/src/routes/notas.routes.ts` — montar rutas bajo `/:id/backlinks`
- `src/backend/src/schemas/backlink.schema.ts` — `CreateBacklinkDtoSchema`, params UUID
- `src/backend/src/errors/` — reutilizar `NotFoundError`, `ValidationError`
- `tests/integration/api/notas.backlinks.test.ts` (nuevo)

**Pasos de implementación:**

1. `POST /:id/backlinks`: validar body `{ destinoId }`; verificar origen y destino existen; rechazar `origenId === destinoId`; `create` en repo; responder `201`.
2. `GET /:id/backlinks/salientes`: join a `notas` destino; devolver `{ id, title }[]`.
3. `GET /:id/backlinks/entrantes`: join a `notas` origen; devolver `{ id, title }[]`.
4. Duplicado → `400` con mensaje «El enlace ya existe».
5. Registrar rutas antes de `/:id` genérico si hay conflicto de orden en Express.
6. Tests integración para ambos escenarios Gherkin a nivel API.

**Criterios de done (verificables):**

- [ ] `POST` crea fila en `nota_backlink`
- [ ] `GET .../entrantes` en destino incluye título de origen
- [ ] `GET .../salientes` en origen incluye título de destino
- [ ] Self-link y duplicado → `400`
- [ ] Nota inexistente → `404`
- [ ] Validación Zod en todos los endpoints

**Tests:**
