## Detalle de implementación (historia)

### Resumen técnico

Extensión de la captura de contenido: el campo `links: string[]` en `CreateNotaDto` se valida con `z.string().url()` en backend y en cliente; al crear/actualizar nota, `NotaService.create` persiste filas en tabla `enlaces` dentro de transacción Prisma. El frontend añade en `NoteForm` filas dinámicas de URL (añadir/eliminar) con feedback inline; en detalle (`NoteDetail`) los enlaces se muestran como `<a>` clicables.

### API involucrada

| Método | Ruta | Uso en esta historia |
|--------|------|----------------------|
| POST | `/api/v1/notas` | Crear nota con array `links` opcional |
| PUT | `/api/v1/notas/:id` | Actualizar enlaces al editar (preparación US-015) |
| GET | `/api/v1/notas/:id` | Verificar enlaces en detalle tras guardar (US-002) |

**Body request con enlaces:**

```json
{
  "title": "Referencia útil",
  "content": "Apuntes con links",
  "links": ["https://ejemplo.com", "https://docs.org/guia"]
}
```

**Respuesta 201 (fragmento links):**

```json
{
  "data": {
    "links": ["https://ejemplo.com", "https://docs.org/guia"]
  }
}
```

**Respuesta 400 (URL inválida):**

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Los datos enviados no son válidos",
    "details": [
      { "field": "links.0", "message": "URL con formato inválido" }
    ]
  }
}
```

### Contratos (DTOs / campos)

| Campo JSON | Tipo | Obligatorio | Notas |
|------------|------|-------------|-------|
| `links` | string[] | no (default `[]`) | Cada elemento: URL válida (`z.string().url()`) |
| `links[].url` (BD) | VARCHAR(2048) | sí si presente | Mapeo inverso en mapper |

Schema: `CreateNotaDtoSchema.links` en `schemas/nota.schema.ts` (LLD §4.3). Mapeo API ↔ BD en `mappers/nota.mapper.ts` §4.4.

## Detalle por task

### TASK-021 — [BE] Validación de URLs y persistencia de enlaces

| Campo | Valor |
|-------|-------|
| **Tipo** | Backend |
| **Release** | MVP |
| **Depende de** | US-005 / TASK-017 (POST /notas); TASK-023 (tabla `enlaces`) |

**Objetivo:** Validar array `links` con Zod y persistir múltiples enlaces por nota en transacción atómica.

**Archivos a crear o modificar:**

- `src/backend/src/schemas/nota.schema.ts` — `links: z.array(z.string().url(...))`
- `src/backend/src/services/nota.service.ts` — extender `create` con persistencia enlaces
- `src/backend/src/repositories/nota.repository.ts` — `createWithRelations` con `enlace.createMany`
- `src/backend/src/mappers/nota.mapper.ts` — `enlaces[].url` → `links[]` orden `created_at` asc

**Pasos de implementación:**

1. Confirmar `CreateNotaDtoSchema.links` con `.default([])` y mensaje ES en `.url()` (LLD §4.3).
2. En `nota.service.create`, tras validación: preparar `links` deduplicados (opcional, negociable).
3. En `nota.repository.createWithRelations`, usar `prisma.$transaction`:
   - `nota.create({ data: { title, content } })`
   - `enlace.createMany({ data: links.map(url => ({ notaId, url })) })` si `links.length > 0`
4. Recargar nota con `include: { enlaces: true, etiquetas: { include: { etiqueta: true } } }`.
5. Mapear respuesta con `toDetail`: URLs ordenadas por `createdAt` asc (LLD §4.4).
6. En `update` (US-015): si `links` presente, delete all + recreate (LLD §5.2 paso 3) — preparar interfaz.
7. Tests: POST con 0, 1, 2 links válidos; POST con `links: ["no-es-url"]` → 400; POST con mix válido/inválido → 400.

**Criterios de done (verificables):**

- [ ] URLs válidas persistidas en tabla `enlaces` con FK correcta
- [ ] URL inválida → 400 `VALIDATION_ERROR` con `field: "links.N"`
- [ ] Respuesta incluye `links[]` reflejando BD
- [ ] Transacción: fallo en enlaces revierte creación de nota

**Tests:**

| Tipo | Ubicación | Qué valida |
|------|-----------|------------|
| Integración | `tests/integration/api/notas.create.links.test.ts` | 0/1/N links, rechazo inválida |
| Unitario | `src/backend/src/schemas/nota.schema.test.ts` | Validación URL Zod |

**Notas técnicas:** Validación URL en capa servicio/Zod, no CHECK regex en BD (BR-005, logical-model §7). VARCHAR(2048) en columna `url`.

### TASK-023 — [DB] Tabla enlaces (nota_id, url) con FK a notas

| Campo | Valor |
|-------|-------|
| **Tipo** | Database |
| **Release** | MVP |
| **Depende de** | TASK-019 (tabla `notas` en misma migración init) |

**Objetivo:** Esquema one-to-many nota → enlaces con cascade delete e índice por `nota_id`.

**Archivos a crear o modificar:**

- `src/backend/prisma/schema.prisma` — modelo `Enlace`
- `src/backend/prisma/migrations/20260704_init_mvp/migration.sql` — DDL tabla `enlaces`
- `src/backend/prisma/seed.ts` — 2 enlaces de ejemplo (logical-model §10)

**Pasos de implementación:**

1. Definir modelo `Enlace` según `logical-model-v1.md` §3.2: `id` UUID PK, `notaId` FK, `url` VARCHAR(2048), `createdAt`.
2. Relación `Nota.enlaces Enlace[]`; `onDelete: Cascade` en FK.
3. Crear índice `idx_enlaces_nota_id` en `(nota_id)`.
4. DDL SQL:
   ```sql
   CREATE TABLE enlaces (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     nota_id UUID NOT NULL REFERENCES notas(id) ON DELETE CASCADE,
     url VARCHAR(2048) NOT NULL,
     created_at TIMESTAMPTZ NOT NULL DEFAULT now()
   );
   CREATE INDEX idx_enlaces_nota_id ON enlaces (nota_id);
   ```
5. Seed: asociar enlaces a notas existentes para pruebas de detalle.
6. Verificar CASCADE: DELETE nota elimina enlaces hijos (BR-004).

**Criterios de done (verificables):**

- [ ] Tabla `enlaces` existe con FK `nota_id` → `notas.id` ON DELETE CASCADE
- [ ] Índice `idx_enlaces_nota_id` presente
- [ ] `schema.prisma` y migración SQL consistentes
- [ ] Seed incluye al menos 1 nota con 2 enlaces

**Tests:**

| Tipo | Ubicación | Qué valida |
|------|-----------|------------|
| Integración | `tests/integration/repositories/enlace.cascade.test.ts` | CASCADE al borrar nota |
