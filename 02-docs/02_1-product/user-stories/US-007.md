## Detalle de implementación (historia)

### Resumen técnico

V1 unifica la experiencia de validación en `NoteForm` (modos create y edit): validación cliente antes de `POST`/`PUT` con mensajes en español por campo; si el servidor responde `400 VALIDATION_ERROR`, el frontend mapea `error.details[]` a errores inline sin resetear `title`, `content`, `links` ni `tags`. El backend ya usa Zod en middleware `validate`; V1 asegura mensajes exactos («El título es obligatorio», «El contenido es obligatorio») y cobertura en create/update y URLs inválidas.

### API involucrada

| Método | Ruta | Uso en esta historia |
|--------|------|----------------------|
| POST | `/api/v1/notas` | Validación create — 400 con `details` |
| PUT | `/api/v1/notas/:id` | Validación update — 400 con `details` |

**Respuesta 400 (validación):**

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Los datos enviados no son válidos",
    "details": [
      { "field": "title", "message": "El título es obligatorio" }
    ]
  }
}
```

### Contratos (DTOs / campos)

| Campo | Validación Zod | Mensaje ES (Gherkin / LLD) |
|-------|----------------|----------------------------|
| `title` | `trim().min(1)` | «El título es obligatorio» |
| `content` | `trim().min(1)` | «El contenido es obligatorio» |
| `links[]` | URL válida por ítem | «La URL no es válida» (US-006, coherente) |
| `tags[]` | strings no vacíos | Mensaje Zod estándar o custom si aplica |

### Archivos principales (según LLD)

### TASK-025 — [BE] Respuestas 400 con detalle por campo

| Campo | Valor |
|-------|-------|
| **Tipo** | Backend |
| **Release** | V1 |
| **Depende de** | US-005 TASK-017 (POST), US-015 TASK-057 (PUT) |

**Objetivo:** Estandarizar respuestas `400 VALIDATION_ERROR` con array `details: { field, message }[]` en create y update de notas.

**Archivos a crear o modificar:**

- `src/backend/src/middleware/errorHandler.ts` — mapeo `ZodError` → `details[]`
- `src/backend/src/schemas/nota.schema.ts` — mensajes `.min(1, "El título es obligatorio")` etc.
- `src/backend/src/middleware/validate.ts` — `validateBody` / `validateQuery`
- `tests/integration/api/notas.create.test.ts` — 400 título/contenido vacíos con `details`
- `tests/integration/api/notas.update.test.ts` — 400 en PUT con mismos mensajes

**Pasos de implementación:**

1. Verificar `CreateNotaDtoSchema` y `UpdateNotaDtoSchema`: mensajes Zod en español alineados con Gherkin.
2. Confirmar `errorHandler` expone `field` como path unido (`title`, `content`, `links.0`).
3. Añadir tests Supertest: POST sin título → 400, `details` contiene `{ field: "title", message: "El título es obligatorio" }`.
4. Repetir para contenido vacío y PUT con body inválido.
5. No devolver 422 ni formatos distintos entre endpoints.

**Criterios de done (verificables):**

- [ ] POST y PUT inválidos → HTTP 400, `error.code === "VALIDATION_ERROR"`
- [ ] `error.details` es array no vacío con `field` y `message` string
- [ ] Mensajes de título/contenido coinciden con Gherkin
- [ ] Tests de integración en verde

**Tests:**

### TASK-026 — [FE] Validación inline y preservación de estado del formulario

| Campo | Valor |
|-------|-------|
| **Tipo** | Frontend |
| **Release** | V1 |
| **Depende de** | TASK-025 (contrato `details` estable) |

**Objetivo:** Mostrar errores junto a cada campo; no resetear estado del formulario tras error cliente o servidor.

**Archivos a crear o modificar:**

- `src/frontend/src/components/notes/NoteForm.tsx` — `fieldErrors`, `validateClient`, render errores por campo
- `src/frontend/src/services/apiClient.ts` — `ValidationApiError` con `fieldErrors: Record<string, string>`
- `src/frontend/src/pages/NoteCreatePage.tsx` — usa `NoteForm` mode create
- `src/frontend/src/pages/NoteDetailPage.tsx` — `NoteForm` mode edit

**Pasos de implementación:**

1. Mantener inputs **controlados** (`useState`); nunca resetear `title`/`content` en catch de error.
2. `validateClient()` antes de submit: si falla, `setFieldErrors` y `return` sin llamar API.
3. Mostrar `{fieldErrors.title}` bajo input título con `role="alert"` o `aria-describedby`.
4. En catch de `ValidationApiError`, mapear `details` a `fieldErrors` sin tocar valores de campos.
5. `noValidate` en `<form>` para controlar validación custom (evitar mensajes nativos del navegador en español inconsistente).
6. Verificar escenario Gherkin: contenido escrito + título vacío → error solo en título, contenido intacto.

**Criterios de done (verificables):**

- [ ] Mensaje «El título es obligatorio» visible junto al campo título
- [ ] Tras error, `content`, `links`, `tags` conservan lo escrito
- [ ] Error 400 del servidor muestra mensajes por campo afectado
- [ ] Comportamiento idéntico en create y edit

**Tests:**

| Tipo | Ubicación | Qué valida |
|------|-----------|------------|
| Unitario | `src/frontend/src/components/notes/NoteForm.validation.test.tsx` | Cliente: título vacío, contenido preservado |
| Unitario | `src/frontend/src/services/apiClient.test.ts` | Parse `details` → `ValidationApiError` |

**Notas técnicas:** Validación en blur es opcional V1; submit es el mínimo exigido por Gherkin.
