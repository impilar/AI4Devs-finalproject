## Historia de usuario

> Como usuario final, quiero buscar notas por un término de texto, para encontrar información sin recordar dónde la guardé.

## Criterios de aceptación (Gherkin)

```gherkin
Feature: Búsqueda de notas

  Scenario: Buscar en título y contenido
    Given que tengo una nota con título "Recetas" y otra con "lista de la compra" en el contenido
    When busco "compra"
    Then veo la nota que contiene "compra" en título o contenido

  Scenario: Búsqueda responde en menos de 300 ms con hasta 500 notas
    Given que existen 500 notas en la biblioteca
    When introduzco un término de búsqueda
    Then los resultados aparecen en menos de 300 milisegundos
```

## Detalle de implementación (historia)

### Resumen técnico

El usuario escribe en `SearchBar` en `HomePage`; tras debounce de 300 ms, `useSearch` llama `GET /api/v1/buscar?q={term}`. `SearchService.search` ejecuta ILIKE insensible a mayúsculas en `title` y `content` (Prisma `contains` + `mode: 'insensitive'`), ordena por relevancia MVP (coincidencia en título antes que solo contenido) y devuelve hasta 50 `NotaResumen`. Los resultados sustituyen temporalmente el listado normal en `NoteList`. Índice `idx_notas_title` y `idx_notas_updated_at` apoyan el SLA RNF-002.

### API involucrada

| Método | Ruta | Uso en esta historia |
|--------|------|----------------------|
| GET | `/api/v1/buscar?q={term}` | Búsqueda en título y contenido |
| GET | `/api/v1/notas` | Restaurar listado al vaciar búsqueda |

**Query params:**

| Param | Tipo | Obligatorio | Default | Notas |
|-------|------|-------------|---------|-------|
| `q` | string | sí | — | `trim`, min 1 carácter (`SearchQuerySchema`) |
| `order` | enum | no | `relevance` | `relevance` \| `date` — ordenación fecha detallada en US-013 |

**Respuesta 200:**

```json
{
  "data": [
    {
      "id": "11111111-1111-1111-1111-111111111102",
      "title": "Lista de la compra",
      "createdAt": "2026-06-12T10:00:00.000Z",
      "updatedAt": "2026-06-12T11:00:00.000Z"
    }
  ],
  "meta": { "q": "compra", "total": 1 }
}
```

**Sin resultados:** `200` con `{ "data": [], "meta": { "q": "...", "total": 0 } }` (mensaje UI en US-014).

**Error 400:** `q` vacío → `VALIDATION_ERROR`.

### Contratos (DTOs / campos)

| Campo | Tipo | Notas |
|-------|------|-------|
| `SearchQuery.q` | string | Obligatorio, trim, min 1 — `schemas/search.schema.ts` |
| `SearchQuery.order` | `relevance` \| `date` | Default `relevance`; post-sort título antes que contenido (LLD §5.5) |
| `NotaResumen` | objeto | Solo campos resumen en respuesta búsqueda |
| `meta.q` | string | Eco del término buscado |
| `meta.total` | number | Cantidad de resultados devueltos (≤ 50) |

## Detalle por task

### TASK-045 — [BE] Endpoint GET /buscar?q= con búsqueda en título y contenido

| Campo | Valor |
|-------|-------|
| **Tipo** | Backend |
| **Release** | MVP |
| **Depende de** | TASK-001 (tabla `notas`); TASK-047 (índices recomendados) |

**Objetivo:** Exponer búsqueda textual ILIKE en `title` y `content` bajo `/api/v1/buscar`.

**Archivos a crear o modificar:**

- `src/backend/src/routes/buscar.routes.ts` — `GET /` → `searchController.searchNotas`
- `src/backend/src/routes/index.ts` — montar router en `/buscar`
- `src/backend/src/controllers/search.controller.ts` — `searchNotas`
- `src/backend/src/services/search.service.ts` — `search(q, order)`
- `src/backend/src/repositories/nota.repository.ts` — `search(term, order)`
- `src/backend/src/schemas/search.schema.ts` — `SearchQuerySchema`
- `src/backend/src/mappers/nota.mapper.ts` — `toResumen` para resultados

**Pasos de implementación:**

1. Definir `SearchQuerySchema` con `q` obligatorio y `order` default `relevance` (LLD §4.3).
2. Registrar ruta `GET /buscar` con `validateQuery(SearchQuerySchema)`.
3. En `nota.repository.search`:

```typescript
await prisma.nota.findMany({
  where: {
    OR: [
      { title: { contains: term, mode: "insensitive" } },
      { content: { contains: term, mode: "insensitive" } },
    ],
  },
  take: 50,
  select: { id: true, title: true, createdAt: true, updatedAt: true },
});
```

4. En `SearchService.search`: si `order === 'relevance'`, post-ordenar (título match antes que solo contenido); si `date`, `orderBy: { updatedAt: 'desc' }`.
5. Responder `200 { data: NotaResumen[], meta: { q, total } }`.
6. `q` vacío tras trim → 400 `VALIDATION_ERROR`.
7. Test integración: búsqueda "compra" encuentra nota por contenido; "Recetas" por título.

**Criterios de done (verificables):**

- [ ] `GET /api/v1/buscar?q=compra` devuelve notas con match en título o contenido
- [ ] Case-insensitive (ILIKE)
- [ ] Máximo 50 resultados (`take: 50`)
- [ ] `meta.q` y `meta.total` presentes
- [ ] Sin lógica Prisma en controller

**Tests:**

| Tipo | Ubicación | Qué valida |
|------|-----------|------------|
| Unitario | `src/backend/src/services/search.service.test.ts` | Orden relevancia, límite 50 |
| Integración | `tests/integration/api/buscar.test.ts` | Contrato HTTP, título/contenido, 400 |

**Notas técnicas:** SQL equivalente en `logical-model-v1.md` §9.3. Full-text GIN evaluable en V1 si ILIKE no cumple RNF-002 en contenido largo.

### TASK-046 — [FE] Barra de búsqueda con resultados en tiempo real

| Campo | Valor |
|-------|-------|
| **Tipo** | Frontend |
| **Release** | MVP |
| **Depende de** | TASK-045 (API buscar) o mock MSW |

**Objetivo:** Input de búsqueda con debounce 300 ms y visualización de resultados en `HomePage`.

**Archivos a crear o modificar:**

- `src/frontend/src/components/search/SearchBar.tsx` — input controlado + debounce
- `src/frontend/src/hooks/useSearch.ts` — estado query, resultados, loading, error
- `src/frontend/src/services/searchApi.ts` — `search(q: string, order?: string)`
- `src/frontend/src/pages/HomePage.tsx` — `SearchBar` + conmutar `NoteList` entre listado y resultados
- `src/frontend/src/components/notes/NoteList.tsx` — reutilizar para mostrar resultados búsqueda

**Pasos de implementación:**

1. Implementar `searchApi.search(q)` → `GET /buscar?q=${encodeURIComponent(q)}`.
2. `useSearch`: estado `query`, `results`, `isSearching`; debounce 300 ms con `useEffect` + timer (LLD §7.3, RNF-002).
3. `SearchBar`: `value`, `onChange`; placeholder "Buscar notas…"; limpiar input restaura listado vía `useNotes`.
4. `HomePage`: si `query.length >= 1` tras debounce, mostrar `results` en `NoteList`; si vacío, listado normal `useNotes`.
5. Indicador loading mientras `isSearching`; `ErrorMessage` en fallo de red.
6. No disparar búsqueda con string vacío (evitar 400).

**Criterios de done (verificables):**

- [ ] Escribir "compra" muestra nota con match en contenido (escenario Gherkin 1)
- [ ] Debounce ≥ 300 ms entre tecleo y request (no un request por tecla)
- [ ] Borrar búsqueda vuelve al listado completo
- [ ] Sin `fetch` directo en `SearchBar` (solo `searchApi`)
